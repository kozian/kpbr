# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project type

kPBR (kozian Policy Based Routing) is a set of POSIX shell scripts and configuration data for **OpenWrt 24.10.2** routers. It implements domain-based split routing: certain hostnames go through a VPN tunnel, others through WAN, decided per-packet via `dnsmasq`-populated `nftset`s + `nftables` packet marking + `iproute2` policy routing tables. There is no build system, no language toolchain, and no test framework — everything is shell + on-router state.

## Top-level scripts (run on the OpenWrt router as root)

- `install-kpbr.sh` — full setup. Reinstalls `dnsmasq-full`, writes `/etc/nftables.d/{sets,rules}.nft`, downloads `nftset.conf` / `vpn-cidrs.lst` / `dns-servers.conf` to their target paths, registers `vpnroute`/`wanroute` in `/etc/iproute2/rt_tables`, generates `/etc/firewall.user`, and installs the hotplug hook `/etc/hotplug.d/iface/25-firewall-user`.
- `update-kpbr.sh` — refreshes `nftset.conf`, `vpn-cidrs.lst`, `dns-servers.conf` from the GitHub `main` branch. Validates with `dnsmasq --test` and re-runs `/etc/firewall.user`; **rolls back** to timestamped `.bak` files if validation fails. Logs to `/var/log/kpbr-update.log`.
- `test-kpbr.sh` — diagnostics only. Dumps `nft` sets, the `mangle_prerouting` chain, `ip rule`, both routing tables, and `dnsmasq --test`.
- `uninstall-kpbr.sh` — removes generated files and rules.
- `create-nftsets.sh` — **runs locally on a dev machine, not on the router**. Regenerates `etc/dnsmasq.d/nftset.conf` from `source/vpn-full.lst` + `source/wan-full.lst`. Use `./create-nftsets.sh` (default with test hosts), `--no-test-hosts`, or pass an output filename as a positional arg.

There are no lint/test commands. Shell syntax can be checked with `sh -n <script>`.

## Architecture: how a packet decides VPN vs WAN

Read this before editing routing logic — the pieces are spread across four files generated by `install-kpbr.sh`:

1. **DNS → set membership.** `dnsmasq-full` reads `/etc/dnsmasq.d/nftset.conf`, where each line is `nftset=/<domain>/4#inet#fw4#<vpn_domain_set|wan_domain_set>`. When a client resolves a listed domain, dnsmasq inserts the answered IPv4 into the matching nftset. Sets are declared in `/etc/nftables.d/sets.nft`.
2. **CIDR seeding.** `/etc/firewall.user` reads `/etc/nftables.d/vpn-cidrs.lst` and pre-populates `vpn_domain_set` with static CIDRs (apple/telegram/etc).
3. **Marking.** `/etc/nftables.d/rules.nft` adds a `mangle_prerouting` chain that sets `meta mark 0x1` for daddr in `vpn_domain_set` and `0x2` for `wan_domain_set`.
4. **Policy routing.** `/etc/firewall.user` adds `ip rule fwmark 0x1 → vpnroute` and `fwmark 0x2 → wanroute`, then populates each table with a default route via the resolved interface. Tables `vpnroute` (100) and `wanroute` (101) are registered in `/etc/iproute2/rt_tables`.

Caveat documented in the README: traffic **originating on the router itself** does not traverse `prerouting`, so `curl` from the router won't be marked. Test from a LAN client.

## Interface resolution (the non-obvious part of `install-kpbr.sh`)

`configure_routing_rules` enumerates `ubus call network.interface dump` and classifies each `up` interface:
- **VPN candidate**: `proto` matches `wgserver|wgclient|wireguard|openvpn|amnezia` OR l3-device matches `amneziawg|awg|wg|tun|sing`.
- **WAN candidate**: has a default route (`route[@.target='0.0.0.0' && @.mask=0]`).

Resolution rules for `VPN_INTERFACE` / `WAN_INTERFACE` (declared at the top of `install-kpbr.sh`):
- Variable set AND matches a candidate → use it.
- Variable empty AND exactly one candidate → auto-detect.
- Zero or multiple candidates → abort with a message asking the user to set the variable.

The generated `/etc/firewall.user` **bakes the resolved interface names** but resolves `l3_device` and `WAN_GW` at runtime via `ubus call network.interface.<name> status`. If an interface is missing at runtime, it logs `[kPBR] ERROR: ... not found` to stderr and **skips just that fwmark rule** — the other side keeps working. `ip route replace` is used so re-runs from hotplug events are idempotent.

The script is `/bin/sh` (busybox ash on OpenWrt) — **do not use bashisms** like `local`, arrays, `[[ ]]`. The exception is `create-nftsets.sh`, which is dev-side `bash`.

## Repo layout

- `etc/dnsmasq.d/{nftset.conf,dns-servers.conf}` and `etc/nftables.d/vpn-cidrs.lst` — these paths mirror their on-router locations; `install-kpbr.sh` and `update-kpbr.sh` download them from `${REPO_URL}${NFTSET_PATH}` etc., so the in-repo path **is** the URL path.
- `source/` — domain/CIDR source lists curated by hand. `vpn-full.lst` and `wan-full.lst` feed `create-nftsets.sh` to (re)generate `etc/dnsmasq.d/nftset.conf`. The `vpn-cidr-*.lst` and `wan-*.lst` files are informational/reference inputs, not directly consumed by any script.
- `manuals/` — Russian-language router-specific install guides (CUDY TR3000/WR3000P, AmneziaWG, kPBR usage). Update these when install-time UX changes.

## Editing conventions observed in this repo

- User-facing strings, log messages, and docs are in **Russian**. Match the existing language when extending them.
- Both `install-kpbr.sh` and `update-kpbr.sh` keep their config block (REPO_URL, *_PATH variables) at the top — keep them in sync if you add a new managed file.
- `update-kpbr.sh` and `install-kpbr.sh` both treat `dnsmasq --test` as the gate before restarting dnsmasq. A bad `nftset.conf` will take down DHCP/DNS, so never skip the test step when adding new dnsmasq configs.
- The `.lst` data files have no comment syntax — `create-nftsets.sh` and `firewall.user` treat every non-empty line as data.
